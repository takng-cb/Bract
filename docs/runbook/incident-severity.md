# 障害重大度 (Severity) 定義と対応目標

## 重大度区分

| Sev | 内容 | 影響範囲の例 |
|---|---|---|
| **Sev1** | 全機能停止 / データ破損 | ログイン不可、ダッシュボードが全顧客で 5xx、DB 全 SELECT エラー |
| **Sev2** | 主要機能停止 | 取引先・商談・物件などの list/detail ページが表示されない、CRUD 不可 |
| **Sev3** | 一部機能異常 / UX 損なう | 特定のフィルタが効かない、ボタンが押せない、表示崩れ |
| **Sev4** | 軽微 | コスメティックな不具合、ログのみ、影響顧客なし |

## 応答時間目標

| Sev | 初回連絡（顧客への 1 次通知） | 復旧目標 (RTO) |
|---|---|---|
| Sev1 | 30 分以内 | 4 時間以内 |
| Sev2 | 1 時間以内 | 1 営業日以内 |
| Sev3 | 1 営業日以内 | 1 週間以内 |
| Sev4 | 翌週リリース時に対応 | 該当なし |

## データ復旧目標 (RPO)

- 全 Sev 共通: **24 時間以内のデータ損失まで許容**（Neon Free plan の PITR 制約）
- Launch plan 以上に移行する場合は 7 日 / 30 日に拡大可能

詳細は [`docs/legal/sla.md`](../legal/sla.md) の 99.5% 月間稼働率保証と整合。

## エスカレーション基準

- Sev1/Sev2 発生 → 直ちに対応開始、原因不明の場合は SNS / 信頼できるコミュニティに技術相談
- 4 時間以内に復旧見込み立たず → 顧客に進捗共有を 1 時間ごとに送る
- 24 時間以内に復旧不可 → SLA 違反として補償対象（[`legal/sla.md`](../legal/sla.md) 参照）

## 障害後

- Sev1/Sev2: [`postmortem-template.md`](postmortem-template.md) を必ず書く
- Sev3: 任意。Issue にコメントとして残せば十分
- いずれも GitHub Issue を起票し、`bug` ラベル + 該当 `priority:*` を付与
