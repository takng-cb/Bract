人材手配業務システム を Bract プラットフォームに落とし込む

結論：人材手配システムは独立した別物ではなく、Bract の staffing 業種モジュール＋AI入力補助(L1) として実装するのが最も自然です。単一企業向け・マルチテナント不要という要件は、Bractの「1社=1デプロイ+1DB」と完全に一致し、CRMコア・粗利計算・AI下書き反映の土台がすでに存在します。本資料はその落とし込み案と、合意が必要な論点を整理したものです。

既存活用 既にBractにある 要拡張 既存を少し広げる 新規 新しく作る 要確認 あなたと合意したい

1. 全体像（結論）

2つの取り組みは競合せず、入れ子の関係になります。

Bract プラットフォーム（CRM/ERP モジュラー基盤）
   ├─ platform（基盤・認証・AI入力補助・監査） ……… 全社共通
   ├─ crm-core / sales / expenses ……………………… CRM 共通
   ├─ erp: inventory / accounting / …（将来）
   └─ industry: real-estate / auto-body /
                └─ ★ staffing（人材手配）← 今回の依頼はここの具体化
                       = 案件(assignments) + 打診(outreach/新規)
                         + 候補(assignment_staff) + 活動 + 予定 + 売上
                         + コピペ→AI解析→確認→反映（AI入力補助 L1）

なぜ噛み合うか ①

単一企業向け・課金不要 = Bractの単一テナント（1社=1デプロイ+1DB）と一致。将来の多社展開も module/build-profile でそのまま開ける。

なぜ噛み合うか ②

clients/agencies/talents/job_orders/candidates/粗利 は 既存 staffing スキーマでほぼ表現済み（後述マッピング）。

なぜ噛み合うか ③

目玉の「コピペ→AI解析→確認→反映」は、設計済みのコントラクトファースト AI 入力補助(draft-then-apply)そのもの。
💡 意思決定の核：「人材手配を Bract の staffing モジュールとして作る」ことに合意できれば、以降の環境・設計・運用は Bract の標準（CLAUDE.md / docs/erp-architecture.md）に乗ります。別物として作る選択肢もありますが、二重投資になるため非推奨です。

4. 要件整理（業務ドメイン）

登場人物

運営会社（ユーザー）
本システム利用者。手配業務を回す。
クライアント／パートナー
案件の発注元。発注単価を払う側。
紹介会社（人材紹介会社）
人材の主供給源。提示単価を受け取る側。
人材
実際にアサインされる人。紹介会社が候補提示。

業務フロー（4フェーズ）＝ 案件ステータスで表現

①受付・起票
LINEコピペ→AI解析→ ②調達・調整
RFQ打診→候補/単価集約→日程調整→ ③確定・実施
アサイン確定→予定→リマインド→ ④売上・請求
粗利確定→月次集計

管理する3本柱＋売上

  • 案件管理：発注1件を軸に打診→候補→確定→実施→完了。既存活用 assignments
  • 活動管理：いつ・誰に・何を／次の一手を時系列。既存活用 activities
  • スケジュール管理：予定＋自動リマインド。要拡張 tasks/新規 events
  • 売上管理：粗利＝発注単価−提示単価、月次・請求/入金。要拡張 既存粗利ロジック＋請求

5. 要件定義（機能・データ）

5.1 MVP 機能 / 画面

#機能状態Bractでの実現
1ログイン（運営スタッフのみ）既存活用Supabase Auth + ロール
2案件ボード（カンバン：受付→…→完了）要拡張商談カンバンUIを案件用に流用
3クイック登録（コピペ→AI→確認→反映）＝目玉新規AI入力補助L1（§6.3）

5スケジュール（カレンダー＋リマインド）新規events + Cron + メール
6売上ダッシュボード（月次/粗利/入金）要拡張forecast/recharts流用＋請求

8設定（Gemini鍵/通知先/リマインド）要拡張/admin/ai 既存＋通知設定

❓ 単価モデルの確認：既存 staffing は「時給×時間（hourly_rate/cost_per_hour）」、ブリーフは「案件あたり固定（client_rate/proposed_rate）」。手配業務は案件固定単価が主と理解。固定単価を主、時給を任意とする方向で良いか？

5.3 目玉：クイック登録のフロー

＋ 新規/情報追加→ 基本情報＋LINE貼付→ [AIで解析] Gemini→ 差分プレビュー
新規=緑/追加=青、手修正可→ [OK] DB反映
✅ intent判定（new_job/update_job/activity_only/schedule_only）、相対日付・金額の正規化、低信頼時の ambiguities 提示 ＝ すべて「コントラクト＋apply層」で表現でき、自動コミット禁止の原則とも一致。

6. 設計

6.1 モジュール配置

src/modules/staffing/
  ├─ manifest.ts          # id:'staffing', category:'industry', deps:['crm-core','sales']
  ├─ contracts.ts         # ★ クイック登録の入力コントラクト（intent/job_order/activities/schedules）
  ├─ schema.ts            # outreach / events / invoices（staffing拡張）→ lib/schema.ts に統合
  ├─ pages/               # 案件ボード, 案件詳細, スケジュール, 売上, マスタ
  ├─ components/          # QuickRegister(貼付→差分プレビュー), CandidateCompare 等
  ├─ actions/             # assignments/outreach/candidates/invoices の server actions(apply層)
  └─ lib/                 # staffingService(粗利・既存), reminderRules, geminiParse

6.2 ゲーティング & セキュリティ（Bract §8 準拠）

  • このデプロイは enabled_modules=['crm-core','sales','staffing']。単一企業なので実質固定。
  • 判定は必ずサーバー側（ensureModuleEnabled / ロール requireEditor）。
  • Gemini鍵はサーバーenvのみ。クライアントへ返さない。設定画面登録時は暗号化保存。
  • 個人情報（氏名・連絡先）はLLM送信を最小化。東京リージョンDB・認証必須。

6.3 AI 入力補助（draft-then-apply）パイプライン

貼付テキスト＋文脈(新規/更新, 対象案件概要, 現在日時)
  → geminiParse()  : contracts.ts のJSON schemaで構造化（厳密JSONのみ）
  → 差分プレビューUI : 新規/更新/活動/予定を色分け表示・項目手修正
  → ユーザー承認
  → apply層(server action) : コントラクト検証 → 既存 createXxx/import経路 → DB
       ★ LLMは直接DB書込みしない。権限・監査(audit_log)を通過
💡 既存の CSV インポート経路と同じ「検証付き書込み」に合流させるため、危険な新規書込み系を作りません。Vertex AI（東京）への切替も lib/ai/providers 抽象で吸収。

6.4 スケジュール & リマインド

  • events(type/start_at/reminder_offsets[ P1D, PT2H ]/reminded)。
  • Vercel Cron が定期実行→対象抽出→メール送信(Resend)。LINE pushは将来。

7. 開発フロー（GitHub運用 / Issue管理）

7.1 リポジトリ & ブランチ

  • リポ：github.com/takng-cb/Bract（CRM履歴統合済み）。旧 Bract-CRM へは push しない。
  • フロー：feature/staffing-* → develop → main（--no-ff）。default直コミット禁止、branch先行。
  • コミット末尾に Co-Authored-By。push/mergeはユーザー指示後。

7.2 Issue 運用（既存AGENTS.md準拠）

  • 不具合/障害は修正コミット前にIssue起票。症状(What)/原因(Why)/修正(How)/検証(Test)/副次タスク。
  • フェーズや機能の塊を umbrella Issue で管理（例：「人材手配MVP」）。
  • Fixes #123 で main マージ時 auto close。close時に本番確認コメント必須。

8. リリース手順

  1. ビルドプロファイル：人材手配顧客は BRACT_BUILD_PROFILE=all（or staffing含む構成）。
  2. 検証ゲート（main反映前・必須）：
    • npm run build（影響する業種env）
    • npm run check:schema（schema↔DB整合・全Neon）
    • npm run test / 既存スモーク・E2E・test:industry-guard
    • 業種/モジュール対称確認（real-estate/auto-body側に波及が無いか）
  3. マージ：feature→develop→main。Vercelが該当projectを再デプロイ。
  4. 実機確認：案件ボード/詳細/クイック登録/スケジュール/売上ダッシュを Chrome で目視。deploy緑をIssueに記録。
💡 機能のON/OFFは再ビルド不要（enabled_modules）。ただし「staffingコードを同梱するか」はビルドプロファイル（再ビルド時のみ）。

9. 顧客追加時の運用手順（将来の多社展開）

今回は1社だが、「将来を閉ざさない」要件に沿って手順を定義。プロビジョニング自動化（ロードマップ Phase6）。

  1. Neon プロジェクト作成（その顧客専用DB）。
  2. 全マイグレーション適用（scripts/apply-migration.ts、冪等）。
  3. マスタ/ライセンス seed：licenses に entitled_modules/enabled_modules=['crm-core','sales','staffing']、object_definitions seed。
  4. Vercel project 作成：Production=main、env（DATABASE_URL/SUPABASE/GEMINI_API_KEY/BRACT_BUILD_PROFILE/RESEND等）。
  5. 初期ユーザー招待（Supabase Auth）＋ロール付与。
  6. 動作確認 → 引き渡し。
