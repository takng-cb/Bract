# ユーザー管理 手順書（招待制）

> Bract のユーザー登録・ロール管理の運用手順。**自動登録は無く、招待制**（REQ-0033）。
> 関連：[`release-neon-rebuild.md`](./release-neon-rebuild.md)（テナント構築）、REQ-0031/ADR-0023（RBAC）。

## 0. 全体像（だれが何をするか）

```
運営（サービス提供者）
  └─ テナント構築時に「初期 admin」を作成（スクリプト）＋ライセンス設定
        ↓
システム管理者（テナント側 admin）
  └─ アプリ内でユーザーを追加・ロール割当・パスワード再発行・削除
        ↓
ユーザー本人
  └─ メール＋初期パスワードでログイン → パスワード変更
     （同じメールの Google アカウントでログインすれば以降 Google で入れる）
```

- **未登録のアカウントはログインできません**（メール／Google どちらの経路でも即サインアウトされ、「システム管理者にユーザー追加を依頼してください」と表示）。

---

## 1. 運営側：初期 admin の作成（テナント構築時に1回）

新しいテナント（Neon）を立ち上げた直後は users テーブルが空で、**誰もログインできません**。
最初の管理者を運営側がスクリプトで作成します。

**前提（.env.local）**
- `DATABASE_URL` … 対象テナントの Neon
- `NEXT_PUBLIC_SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY` … Auth ユーザー作成に必要（秘匿。コミット禁止）

**実行**
```powershell
npx tsx scripts/create-admin-user.ts admin@example.com 'Strong-Password-Here'
```

- 出力の `📍 対象 DB:` が意図したテナントの Neon か必ず確認すること。
- Supabase Auth に同じメールが既に居る場合は再利用して users 行のみ admin 化（冪等）。
- RBAC の roles が seed 済みなら admin ロール（role_id）も自動割当。

> ライセンス（プラン・enabled/entitled モジュール）の設定も運営側の作業。
> `/admin/license`（提供者向けセクション）または licenses 行を直接投入。

---

## 2. 管理者：ユーザーの追加

1. 管理者でログイン → サイドバー下部 **「システム設定」**（`/settings/system`）
2. **「ユーザー管理」** セクションで入力して追加：
   - メールアドレス（本人が使うもの。**Google 紐づけしたい場合は Google アカウントと同じメール**にする）
   - 初期パスワード（8文字以上）
   - ロール：**閲覧者（viewer）／編集者（editor）／管理者（admin）**
3. 追加した瞬間に有効（確認メールは送信されません）
4. 本人に「ログイン URL＋メール＋初期パスワード」を安全な経路で伝える

**ロールの意味（標準3ロール）**

| ロール | できること |
|---|---|
| admin | 全権（設定・ユーザー管理含む） |
| editor | 全ブックの作成・閲覧・更新・削除 |
| viewer | 全ブックの閲覧のみ |

**ブック単位で細かく制御したい場合（RBAC）**
1. `/admin/roles` で **カスタムロール**を作成（例：営業／経理／工場）
2. ブック×「作成/閲覧/更新/削除」のマトリクスをチェックで設定
   - 「すべてのブック（既定）」行＝既定値、ブック行で上書き
   - **閲覧 OFF のブックは、その人のナビ・検索・一覧・詳細から消えます**
3. 同画面の「ユーザーへのロール割当」で割当（反映まで最大60秒）

---

## 3. ユーザー本人：初回ログインと Google 紐づけ

1. ログイン画面でメール＋初期パスワードでログイン
2. 「設定」（`/settings`）からパスワードを変更
3. **Google でログインしたい場合**：ログイン画面の「Google でログイン」を、**登録されたメールと同じ Google アカウント**で実行
   - 検証済みメール同士は Supabase が自動でアカウントを紐づけます（リンク完了）
   - 以降はメール／Google のどちらでもログイン可能
   - ⚠️ **別のメールの Google アカウント**でログインすると「登録されていません」と拒否されます（仕様）

---

## 4. 管理者：日常運用

| 操作 | 場所 |
|---|---|
| ロール変更 | `/settings/system` のユーザー管理（標準3ロール）、`/admin/roles`（カスタムロール割当） |
| パスワード再発行 | `/admin/users` |
| ユーザー削除 | `/admin/users` |
| 代理ログイン（なりすまし） | `/settings/system` のユーザー管理 →「このユーザーとしてログイン」→ 画面下のバナーから管理者に復帰 |

- ユーザー数の上限はライセンス `features.max_users` で制御（未設定=無制限）。

---

## 5. トラブルシュート

| 症状 | 原因と対処 |
|---|---|
| 「このアカウントは登録されていません」 | 招待されていない（または別メールの Google）。管理者がユーザー追加するか、登録済みメールと同じ Google アカウントを使う |
| Google ログイン後にログイン画面へ戻る | 旧バージョンの既知バグ（#44、修正済み）。再発したらブラウザの Cookie 削除＋最新デプロイを確認 |
| 管理者が誰もいない／ログインできない（ロックアウト） | 運営側で §1 の `create-admin-user.ts` を再実行（既存メールでも admin に upsert される） |
| ロール変更が反映されない | 最大60秒のキャッシュあり。それ以上続く場合は再ログイン |
| 旧「メンバー」ロールのユーザーが編集できない | 旧仕様の値（member）は閲覧のみ扱い。ユーザー管理でロールを選び直して保存 |
