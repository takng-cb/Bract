# 人材手配業務 効率化システム — Claude Code 開発ブリーフ

> このドキュメントは Claude Code への開発依頼書です。リポジトリ直下に `CLAUDE.md` として置くか、最初のプロンプトとして貼り付けて使ってください。まず全体を読み、不明点は実装前に質問してください。いきなり全機能を作らず、後半の「開発フェーズ計画」に沿って段階的に進めてください。

---

## 1. プロジェクト概要

人材手配・マッチング業を営む**1社（運営会社）専用**の業務管理Webシステムを開発する。現状 Excel ＋ LINE ＋ 口頭管理で分散している業務を、ひとつのシステムに集約する。

- **発注者（依頼主）**：このシステムを使う運営会社。クライアントから案件を受け、紹介会社経由で人材をあてがう。
- **収益構造**：発注単価（クライアントからもらう）と 提示単価（紹介会社へ払う）の差額＝粗利。
- **本システムは単一企業向け**。マルチテナント・サブスク課金は不要。ただし将来の多社展開を完全に閉ざさないよう、データ構造は素直に保つ。
- **狙いの核**：LINEのやりとりを「コピペ→AI解析→確認→反映」で構造化し、転記作業をゼロにする。

---

## 2. 業務ドメイン（理解の土台）

### 登場人物
- **運営会社（ユーザー）**：本システムの利用者。手配業務を回す。
- **クライアント／パートナー**：案件の発注元（例：「明日2万で●●の人材いない？」「◯日〜◯日で空いてる人いる？」）。
- **紹介会社（人材紹介会社）**：人材の主な供給元。**人材は紹介会社経由が中心**。
- **人材**：実際にアサインされる人。紹介会社が候補として提示してくる。

### 業務フロー（4フェーズ）
1. **受付・起票**：クライアントからLINEで発注 → コピペしてAI解析 → 案件として起票。
2. **調達・調整**：複数の紹介会社へ打診（RFQ型）→ 候補＋提示単価を集約 → 単価比較。打ち合わせ・面談の日程調整。
3. **確定・実施**：候補を確定（アサイン）→ 実施日をスケジュール登録 → 関係者へリマインド配信。
4. **売上・請求**：発注単価と提示単価から請求・支払・粗利を確定 → 月次で集計。

### 管理する3本柱 ＋ 売上
- **案件管理**：発注1件1件を軸に、打診・候補・確定・実施・完了を管理。
- **活動管理**：いつ・誰に・何をしたか／次の一手、を時系列で記録。
- **スケジュール管理**：打ち合わせ・案件実施・返信期限などの予定＋自動リマインド。
- **売上管理**：案件ごとの粗利と月次売上・請求/入金ステータス。

---

## 3. 技術スタック & 前提

| 項目 | 採用 | 補足 |
|---|---|---|
| フレームワーク | **Next.js（App Router）+ TypeScript** | strict有効 |
| UI | Tailwind CSS（+ shadcn/ui 任意） | 日本語UI。モバイル/PC両対応（レスポンシブ or PWA） |
| DB / 認証 / ストレージ | **Supabase（東京リージョン）** | Postgres + Auth + RLS |
| ホスティング | **Vercel** | Cron 機能をリマインドに利用 |
| LLM | **Gemini API** | **APIキーは運営会社（先方）が用意**。サーバー側のみで利用し、クライアントに露出させない |
| LINE連携 | **当面なし（コピペ運用）** | 将来 LINE公式アカウント連携へ拡張可能な設計に留める |

- **LINE API は今は実装しない**。すべてユーザーが文面を貼り付け、LLMで解析する。
- Gemini APIキーは**サーバー環境変数**として保持（Route Handler / Server Action 内でのみ使用）。設定画面から登録する方式にする場合も、暗号化して保存し、クライアントへは絶対に返さない。

## 4. 主要機能 / 画面（MVPスコープ）

1. **ログイン**（Supabase Auth、運営スタッフのみ）
2. **案件ボード**：ステータス列（受付 → 打診中 → 候補集約 → 確定 → 実施 → 完了）でカンバン表示
3. **クイック登録**（=目玉機能。次章参照）：画面上部の目立つ「＋」から起動
4. **案件詳細**：案件情報 ＋ 打診状況（紹介会社別）＋ 候補一覧（提示単価・粗利の比較）＋ 活動タイムライン ＋ 関連予定
5. **スケジュール**：カレンダー表示＋「今日／今週」＋リマインド設定
6. **売上ダッシュボード**：月次売上・粗利、請求/入金ステータス
7. **マスタ管理**：クライアント／紹介会社／人材
8. **設定**：Gemini APIキー、通知先（メール）、リマインドルール

## 5. 目玉機能：クイック登録（コピペ → AI解析 → 確認 → 反映）

最重要のUX。以下の流れで実装する。**LLMの出力は必ず確認画面を経てから書き込む（自動コミット禁止）**。

```
[新規作成] or [情報追加] ボタン
   ↓
基本情報入力 ＆ LINE貼り付け画面
   （クライアント等の任意項目 ＋ LINE本文テキストエリア）
   ↓
[AIで解析] → Gemini が構造化（intent判定：新規/更新/活動のみ/予定のみ）
   ↓
情報更新の確認画面（差分プレビュー）
   ・案件：新規作成 or 既存案件の更新
   ・活動ログ：追加
   ・スケジュール：追加（リマインド設定込み）
   ↓
[OK] → DBへ反映
```

- 「新規作成」と「情報追加（既存案件への追記）」の両方に対応。更新時はAIが対象案件を推定し、確認画面で対象を選べるようにする。
- 確認画面は**差分が一目で分かるUI**（新規=緑、追加=青 等のバッジ）。各項目は手修正してから反映できる。

## 6. データモデル

Supabase マイグレーションで定義する。主要テーブルと要点：

| テーブル | 主なカラム |
|---|---|
| `clients`（クライアント） | name, contact_person, line_type(個人/公式), notes |
| `agencies`（紹介会社） | name, contact, specialties, notes |
| `talents`（人材／任意・リピート用） | name, agency_id(null可=自社), skills, default_rate, contact |
| `job_orders`（案件） | client_id, role, headcount, work_date, start_time, end_time, location, **client_rate(発注単価)**, status, raw_message, notes |
| `outreach`（打診） | job_order_id, agency_id, status(打診済/返信待ち/候補あり/該当なし), sent_at |
| `candidates`（候補） | job_order_id, agency_id, talent_name, talent_id(null可), **proposed_rate(提示単価)**, status(候補/確定/辞退) |
| `activities`（活動） | related_type(案件/クライアント/紹介会社), related_id, type(打診/LINE受信/電話/候補受領/確定/メモ等), occurred_at, body, next_action |
| `events`（予定） | job_order_id(null可), type(打ち合わせ/案件実施/返信期限/その他), title, start_at, end_at, attendees, reminder_offsets, reminded |
| `invoices`（請求・売上） | job_order_id, candidate_id, billing_amount(=発注単価), payment_amount(=提示単価), **margin**, billing_status, payment_status, billed_at, paid_at |

- `job_orders.status` で4フェーズの進行を表現。
- **粗利 = 発注単価 − 提示単価**。確定候補から自動計算し、`invoices` と売上集計で参照。
- 単一企業向けだが、全テーブルに RLS を有効化し「ログイン済みスタッフのみ参照・編集可」を基本ポリシーとする。

## 7. Gemini 解析の仕様

- **入力**：貼り付けたLINE本文 ＋ コンテキスト（新規/更新の別、更新時は対象案件の概要、現在日時）。
- **出力**：**厳密なJSONのみ**（前置き・マークダウン無し）。例の構造：

```json
{
  "intent": "new_job | update_job | activity_only | schedule_only",
  "target_job_hint": "更新時の対象案件の手がかり（無ければ null）",
  "job_order": { "role": "...", "headcount": 1, "work_date": "2026-06-09",
    "start_time": "09:00", "end_time": "18:00", "location": "梅田",
    "client_rate": 20000, "client_name": "○○商事" },
  "activities": [{ "type": "LINE受信", "occurred_at": "...", "body": "..." }],
  "schedules": [{ "type": "案件実施", "title": "...", "start_at": "...", "reminder_offsets": ["P1D"] }],
  "ambiguities": ["曖昧だった点（あれば）"]
}
```

- 相対日付（「明日」）は現在日時を基準に絶対日付へ解決。金額表現（「2万」）は数値へ正規化。
- 解析結果は必ず確認画面に出し、ユーザー承認後にDB書き込み。失敗・低信頼時は `ambiguities` を表示して手修正を促す。
- **個人情報配慮**：氏名・連絡先など個人情報は最小限のみ送信。コンプライアンス重視なら AI Studio キーではなく **Vertex AI（東京リージョン）** 利用を検討（実装は切替可能な抽象化を挟む）。

---

## 8. スケジュール & リマインド

- `events` に打ち合わせ・案件実施・返信期限を登録。`reminder_offsets`（例：前日 `P1D`、2時間前 `PT2H`）でリマインドルールを持つ。
- **Vercel Cron**（または Supabase scheduled function）で定期実行し、対象予定を抽出して通知。
- 通知チャネルは**まずメール**（Resend 等の送信サービス、無料枠で開始可）。**LINE push は将来拡張**（LINE公式連携時）。アプリ内通知も可。

---

## 9. 売上・請求

- 確定時に `invoices` を生成（請求額＝発注単価、支払額＝提示単価、粗利を確定）。
- 請求/入金ステータスを管理し、月次の売上・粗利を集計・ダッシュボード表示。
- 会計ソフト（freee／マネーフォワード）連携は将来検討（今はCSVエクスポートで十分）。

---

## 10. 非機能要件

- **言語**：UIは日本語。
- **個人情報**：人材の氏名・連絡先を扱う。DBは東京リージョン、認証必須、RLSで保護。LLM送信時は個人情報を最小化。
- **セキュリティ**：Gemini APIキー等のシークレットはサーバー側のみ。クライアントへ露出しない。フォーム入力はバリデーション。
- **品質**：TypeScript strict、エラーハンドリングとローディング状態を丁寧に。

---

## 11. 開発フェーズ計画

各フェーズ完了ごとにコミットし、レビューを受けてから次へ進む。

- **Phase 0｜環境構築**：Next.js(App Router)+TS、Tailwind、Supabase接続、Vercel、環境変数、基本レイアウト。
- **Phase 1｜認証 + スキーマ + マスタ**：Supabase Auth、全テーブルのマイグレーション＋RLS、クライアント/紹介会社/人材のCRUD。
- **Phase 2｜案件管理**：案件CRUD、案件ボード、案件詳細、打診・候補（単価比較・粗利計算）。
- **Phase 3｜クイック登録（目玉）**：貼り付け→Gemini解析→差分確認→反映。新規/更新の両対応。
- **Phase 4｜活動 + スケジュール + リマインド**：活動タイムライン、events、Cronによる通知（メール）。
- **Phase 5｜売上・請求**：invoices、請求/入金管理、月次ダッシュボード。
- **Phase 6｜仕上げ**：UI/UX調整、日本語コピー、バリデーション、空状態・エラー表示。

---

## 12. 作業の進め方（Claude Code へのルール）

- 大きな設計判断（依存追加、スキーマ変更、外部サービス選定）は**実装前に確認**する。
- スキーマ変更は必ず**マイグレーションファイル**で行う（手作業でのDB直編集をしない）。
- **LLMの出力は必ず確認画面を経てから書き込む**（自動コミット禁止）。
- シークレットをコードやクライアントに埋め込まない。`.env.example` を用意。
- フェーズ単位で小さくコミットし、各フェーズ末に動作確認手順を示す。
- ユーザー（依頼主）の役割は**コードレビューとプロダクト判断**。実装はClaude Codeが主導する。
